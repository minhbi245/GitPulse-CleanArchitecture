# Code Standards — GitPulse iOS

## Swift Style

- Swift 5.9+, no Objective-C interop required beyond UIKit/Core Data
- Prefer `struct` for domain models, `final class` for ViewModels, services, and repositories
- Use `enum` with no cases as namespaces (e.g., `AppColors`, `AppTypography`, `APIEndpoint`)
- Mark all `@MainActor` on ViewModels; use `nonisolated` on init when concurrency allows
- Prefer `[weak self]` captures in closures; use `guard let self` pattern
- Use `defer` for cleanup (loading state, isLoading guard resets)
- No force unwraps (`!`) except Asset Catalog colors that are guaranteed present — and even those use `?? fallback`

---

## Naming Conventions

### Files

- PascalCase for all Swift files matching the type they contain: `UserListViewModel.swift`, `AppColors.swift`
- Test files: `{TypeName}Tests.swift` — e.g., `GetUserPagingUseCaseTests.swift`
- Mock files: `Mock{ProtocolName}.swift` — e.g., `MockUserRepository.swift`
- Core Data generated files: `{Entity}+CoreDataClass.swift`, `{Entity}+CoreDataProperties.swift`

### Types

| Kind | Convention | Example |
|------|-----------|---------|
| Struct/Class/Enum | PascalCase | `UserModel`, `PaginationManager` |
| Protocol | PascalCase + suffix `Protocol` | `UserRepositoryProtocol`, `UserServiceProtocol` |
| Enum case | camelCase | `.noConnection`, `.loadingMore` |
| Property | camelCase | `avatarUrl`, `lastUserId` |
| Method | camelCase verb | `loadInitialData()`, `showUserDetails(username:avatarUrl:url:)` |
| Constant (static let) | camelCase | `pageSize`, `cacheTimeout` |

### Protocols

Suffix protocol names with `Protocol` (not the `able`/`ing` pattern) to clearly distinguish from concrete types: `UserRepositoryProtocol` vs `UserRepositoryImpl`.

---

## File Size

Maximum 200 LOC per Swift file. When a file exceeds this:
1. Extract sub-components into separate files (e.g., UIView components to `Components/` subfolder)
2. Split extensions by concern using `// MARK: -` sections first, then separate files if still large
3. Extract mappers to dedicated `Mappers/` files

## Clean Architecture Layering Rules

### Dependency Direction

```
Presentation --> Domain <-- Data
```

- Domain imports: Foundation only (no UIKit, no Combine, no third-party)
- Data imports: Foundation, Combine, Core Data, Factory, and Domain
- Presentation imports: UIKit, Combine, Factory, and Domain (never imports Data types directly)

### Enforcement Checks

Before adding an import, verify:
- Domain file adding `import UIKit` → violation, reject
- Data file adding `import UIKit` → violation (exception: Core Data entity subclasses generated by Xcode)
- Presentation file importing a concrete Data type (not via protocol) → violation

### Concrete Type Access in Presentation

ViewModels receive use cases (Domain) via Factory injection. The ViewModel never instantiates `UserRepositoryImpl` or `UserService` directly. Factory wires concrete types at the `DataContainer` level.

```swift
// Correct: inject via Factory
@Injected(\.getUserPagingUseCase) private var useCase

// Correct: convenience init uses Container.shared
convenience init() {
    self.init(paginationManager: Container.shared.paginationManager())
}

// Wrong: direct instantiation in ViewModel
let repo = UserRepositoryImpl(...)
```

## Combine Patterns

### Publishers

- Use `AnyPublisher<Output, Error>` as the return type at protocol boundaries
- Wrap `async throws` calls in `Future<T, Error>` inside repository methods
- Use `eraseToAnyPublisher()` before returning from any public method

```swift
func getUsers(perPage: Int, since: Int) -> AnyPublisher<[UserModel], Error> {
    Future<[UserModel], Error> { promise in
        Task {
            do {
                let result = try await self.userService.getUsers(perPage: perPage, since: since)
                promise(.success(result))
            } catch {
                promise(.failure(NetworkErrorMapper.mapError(error)))
            }
        }
    }.eraseToAnyPublisher()
}
```

### Subjects

- `CurrentValueSubject` for state that replays the latest value to new subscribers
- `PassthroughSubject` for one-shot events with no replay (navigation, toasts, alerts)
- Annotate subjects with `@MainActor` when they drive UI updates

### Subscriptions

- Store all subscriptions in `Set<AnyCancellable>` (inherited from `BaseViewModel.cancellables`)
- Use `.sink { [weak self] in ... }.store(in: &cancellables)` pattern
- Never retain `AnyCancellable` as a property without storing in a set

### removeDuplicates

Apply `.removeDuplicates()` on state publishers when `UiState` conforms to `Equatable` to prevent redundant UI updates.

---

## async/await Patterns

Use `async throws` for networking calls in the Service layer. Do not use Combine at the service level — convert at the repository layer via `Future`.

```swift
// Service: async/await
func getUsers(perPage: Int, since: Int) async throws -> [UserResponse] {
    try await apiClient.request(.getUsers(perPage: perPage, since: since))
}

// Repository: wraps in Future -> AnyPublisher
// (see Combine Patterns above)
```

Use `Task { @MainActor in ... }` when dispatching back to the main actor from a `Task` created in a non-isolated context.

## Dependency Injection (Factory)

All registrations live in `DataContainer.swift` (`extension Container`). `AppContainer.swift` exists as an extension placeholder — add app-level overrides there if needed.

### Lifetimes

| Lifetime | Usage | Example |
|----------|-------|---------|
| `.singleton` | Shared state, one instance per app | `apiClient`, `coreDataManager`, `paginationManager` |
| (default transient) | Fresh instance per injection | `getUserPagingUseCase`, `getUserDetailsUseCase` |

### Injection in ViewModels

Prefer the convenience `init()` pattern that calls `Container.shared` so that:
1. Unit tests can inject mocks via the primary `init`
2. App code uses the default convenience `init` with no arguments

```swift
// Primary init (testable)
init(paginationManager: PaginationManager) { ... }

// Convenience init (app entry point)
convenience init() {
    self.init(paginationManager: Container.shared.paginationManager())
}
```

## ViewModel Conventions

- All ViewModels are `@MainActor final class` extending `BaseViewModel<UiState, Event>`
- `UiState` is a `struct` conforming to `Equatable`
- `Event` is an `enum` conforming to `Equatable` for one-shot navigation/action events
- Actions are named as verbs: `loadInitialData()`, `refresh()`, `selectUser(_:)`
- Use `performTask(showLoading:onError:task:)` from `BaseViewModel` for async work with automatic error and loading state management

---

## ViewController Conventions

- No business logic in ViewControllers — only UI binding and delegation
- `viewDidLoad`: call `setupUI()`, `setupBindings()`, `viewModel.loadInitialData()`
- Separate `setupUI()` (layout/style) from `setupBindings()` (Combine subscriptions)
- Assign `coordinatorDelegate` (weak) for navigation rather than pushing VCs directly
- Use `// MARK: - SectionName` to organize: Setup, Bindings, Actions, Delegate conformances

---

## Testing Conventions

- Test files mirror source structure: `GitPulseTests/Domain/`, `GitPulseTests/Data/`, `GitPulseTests/Presentation/`
- Mocks in `GitPulseTests/Mocks/` implement the same protocols used in production code
- Use `XCTestExpectation` + `waitForExpectations` for async Publisher tests
- Name tests: `test_{scenario}_{expectedOutcome}` — e.g., `testLoadInitialData_success_updatesUserList`
- No network calls in unit tests — all tests use mock service/repository

---

## Error Handling

- All errors at the Domain boundary are `AppError` (mapped by `NetworkErrorMapper.mapError(_:)`)
- ViewModels call `showError(_:)` (from `BaseViewModel`) which maps to `ErrorState` via `ErrorStateMapper.map(_:)`
- ViewControllers present alerts via `ErrorAlertHelper` in response to `errorPublisher`
- Append-load errors (non-empty list, non-refreshing) render inline in `LoadMoreFooterView`, not as alerts

---

## UI Layout

- All layout via SnapKit (`snp.makeConstraints`, `snp.updateConstraints`, `snp.remakeConstraints`)
- No storyboards or XIBs (except `LaunchScreen.storyboard`)
- Use `UIFont.preferredFont(forTextStyle:)` + `adjustsFontForContentSizeCategory = true` for all text labels
- Use `UIColor(named:)` from Asset Catalog for all colors; never hard-code hex values
- Minimum touch target: 44pt height/width for all interactive elements

---

## MARK Organization

Standard section order within a Swift file:

```swift
// MARK: - Properties
// MARK: - Init
// MARK: - Lifecycle (UIViewController only)
// MARK: - Public API
// MARK: - Actions
// MARK: - Private
// MARK: - {Protocol Name} (one per conformance)
```
